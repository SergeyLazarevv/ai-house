# Сценарий: расследование недоставленного SMS

Когда пользователь спрашивает, почему не доставлено SMS, оркестратор обычно действует так:

1. Сначала `db`.
Задача для БД: найти пользователя по номеру телефона или другому идентификатору, при необходимости связать `phone` и `user_id` через таблицу `users`.
Нужно вернуть только факты, полезные для следующих шагов: `user_id`, `phone`, `is_blocked`, `message_id`, `owner_id`/`user_id`, `provider`, `status`, `error_code`, `created_at`.

2. Затем `logs`.
Задача для логов: искать ошибки отправки или доставки вокруг `created_at`, используя `user_id`, `message_id`, `owner_id`/`user_id`, `provider`, `phone` и другие найденные идентификаторы.
Ищи ERROR/WARN и явные причины: `timeout`, `provider reject`, `invalid destination`, `insufficient balance`, `rate limit`, `validation error`.

3. Выбирай `finish`, если причина уже понятна из БД и логов.
Не нужно вызывать `code`, если есть ясная прикладная или провайдерная причина.

4. Вызывай `code` только если логи указывают на баг приложения.
Признаки: stack trace, имя файла/класса/метода, повторяющаяся внутренняя ошибка сервиса без понятной внешней причины.

5. Для исполнителей передавай только краткие факты.
В `context_hint` передавай только идентификаторы и период поиска, например:
`message_id=123, owner_id=456, provider=smsc, created_at=2026-04-01T10:00:00Z`.
